03 What's actually different

Three properties we haven't seen combined in another public music AI system: diffusion-native temporal control, high-throughput streaming, and live LoRA refit.

i.

Per-frame control of the diffusion process

Several solver-side controls that are normally single numbers become curves drawn over the song's timeline, at the same 25 Hz resolution as the latent itself. Source preservation oscillates rhythmically. Transformation pressure rises through the chorus. The engine exposes six temporal curves; the per-step curves can update in-flight through shared state.

ii.

12.3 generations per second

A streaming ring-buffer pipeline keeps 8 in-flight 60-second song generations at staggered denoising stages, batched into a single forward pass per tick. After warmup, one finished 60-second song latent emerges every 81 ms. The user hears a moving window through a song that's continuously being re-drafted.

iii.

Hot-swap LoRAs into a live TensorRT engine

TensorRT engines are normally static weights baked at build time. DEMON refits LoRAs into the running engine in ~1.2 seconds, with no rebuild. Drop a .safetensors file in the LoRA folder and play it through the model that's currently making sound.

05 How it works

DEMON is not the music model. It's a streaming runtime, control surface, and acceleration layer built around ACE-Step v1.5, an open, MIT-licensed music diffusion model from the ACE-Step team. Without their model, none of this exists.

What the model sees: a 1500 × 64 latent

ACE-Step's convolutional VAE compresses 60 seconds of 48 kHz stereo audio into a single tensor of shape [1500, 64]: 1500 frames at 25 Hz, 64 channels each. That tensor is the unit of generation. A Diffusion Transformer (DiT, 2 B parameters) refines it from pure noise to clean music over 8 denoising steps using flow matching.

For an audience used to image diffusion, the mental anchor is precise: a 60-second song behaves like a single latent image, 1500 pixels wide, 64 channels deep. Many familiar image-diffusion ideas have audio-latent counterparts here: masking and inpainting, conditioning, LoRAs, classifier-free guidance, and temporal control curves, all on a one-dimensional time axis instead of a two-dimensional spatial one.

The streaming pipeline: ring buffer over depth N

A naïve real-time loop would run all 8 denoising steps for one song, output it, then start the next, capping at one song every (8 × tick_ms). DEMON adapts StreamDiffusion's ring-buffer pattern to audio: 8 in-flight 60-second song generations are kept at staggered denoising stages, batched into a single forward pass per tick. After a warmup period, one finished song emerges every tick. 12.3 per second on a 5090.

The user doesn't hear 12 separate songs per second. They hear a moving playhead window through one of these latents, decoded as audio, while newer latents continuously evolve in the buffer behind it. The future of the song is being continuously re-drafted until the moment the playhead reaches each frame. That property is what makes per-frame control land. There's still a "future" to influence, even mid-playback.

Compare autoregressive systems: once a chunk is committed audio, that chunk is locked. Compare batch generation: by the time you hear it, the parameter choices were frozen before render. DEMON's streaming-diffusion shape keeps the still-evolving latent generation and the active control surface in contact, at diffusion-frame resolution.

Per-frame curves: the centerpiece contribution

The most musically impactful diffusion parameter, denoise (the strength of the AI remix), is global in the standard ACE-Step scheduler. It works by truncating the slot's timestep schedule, and the model forward pass advances every frame in that slot at the same timestep. You cannot have one part of a song heavily transformed and another part lightly transformed using denoise alone. In this scheduler path, it is a global scalar by construction.

DEMON recovers denoise-like musical control with an SDE solver variant that works through a different mechanism. At every denoising step, the engine computes two re-noising candidates per frame, one anchored to the model's prediction (generation), one anchored to the source latents (preservation), and blends them according to a per-frame curve that can vary across all 1500 frames of the song. Because the blending happens after the model's forward pass and is re-decided at every step, different frames can take different effective source-preservation strengths without changing the scheduler's timestep contract.

Six temporal controls are exposed in total: source-preservation (the SDE curve above), velocity scale (transformation rate), guidance scale (per-frame CFG), ODE noise injection, initial-noise mixing, and x0-target morphing. They operate on the same temporal axis and compose through the streaming pipeline; the per-step controls are the ones that can bypass ring-buffer drain via shared mutable state.

Shared mutable curves: 1-tick latency

Per-request parameters (text prompts, source audio, the denoise scalar) are baked into a slot at submission time and only manifest after the ring buffer drains (~649 ms at depth 8). Per-step parameters such as SDE source-preservation, velocity scale, ODE noise, and x0-target strength can be read fresh from shared state at every denoising step. When one of those shared curves is updated, all 8 in-flight slots use the new value on the very next tick. That's ~81 ms knob-to-latent at depth 8, or ~102 ms to decoded audio with the windowed VAE.

This is the property that makes the system feel less like a render queue and more like a responsive musical surface. Below the threshold of human-perceptible lag, gesture and sound feel coupled.

Acceleration: TensorRT mixed precision + LoRA refit

The DiT is exported to ONNX and compiled to TensorRT with a careful precision split: attention and MLP blocks in fp16 for speed, but the adaptive normalization (AdaLN) pathway, timestep embeddings, and RMSNorm layers remain in fp32. Without this, fp16 quantization compounds across 24 layers, measured at ~7× attenuation of velocity output and NaN failures at early timesteps. With it, the engine runs cleanly.

TensorRT engines are normally static. DEMON enables the REFIT builder flag and exports ONNX with constant folding disabled, preserving PyTorch parameter names as ONNX initializer names. At runtime, the IRefitter API computes LoRA deltas and applies them to the live engine. No rebuild, ~1.2 seconds, ~8% inference overhead.

Windowed VAE decode: 3× speedup, bit-identical

Full VAE decode of a 60-second latent costs ~63 ms, nearly as much as the DiT forward pass itself. But during streaming, the user only hears a few seconds at a time. The Oobleck VAE uses dilated 1D convolutions whose receptive field, empirically, converges within 333 ms of any boundary. Decoding the playback window plus 0.5 s of overlap on each side produces output that is bit-identical to the full decode in the interior. 21 ms instead of 63 ms, no quality loss.

06 Run it

Python 3.11, a CUDA GPU, and uv. ACE-Step v1.5 checkpoints download automatically on first run. 

# clone
git clone https://github.com/ryanontheinside/DEMON.git
cd DEMON

# install
uv sync

# launch the realtime web demo
uv run python -u -m demos.realtime_motion_graph_web --port 8765

Then open http://localhost:8765/. Drop in any .wav as source, type a prompt, twist a knob (or wire a MIDI controller via Web MIDI), and listen.

For LoRA-conditioned generation, drop .safetensors files in ~/.daydream-scope/models/demon/loras/. Two known-good ones to start with: synthpop and the deathstep Civitai community LoRA.
